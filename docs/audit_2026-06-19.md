# PaySpyre Backend ‚Äî Full Code Audit

**Date:** 2026-06-19
**Branch audited:** `feature/p8.0-observability` (HEAD `78fb47f`)
**Scope:** Backend (`payspyre-backend`) only. **No frontend exists on the build machine** ‚Äî see Finding F-0.
**Method:** Six parallel auditors read the actual implementation (not docs/commit messages), verified each claim against code, and ran the test suite. Findings carry `file:line` evidence.

---

## 0. Executive summary

The **new platform code** (flow orchestrator, applicant API, magic-link auth, vendor webhooks, real notification adapters, PostHog observability) is genuinely well-built and, in its own lane, secure. The problems are:

1. A **legacy V1 lending surface was still mounted and completely unauthenticated** ‚Äî able to approve/reject/fund loans, pull bureau credit reports, and stream the entire loan book as CSV, all with no auth. **(FIXED 2026-06-19 ‚Äî see ¬ß6.)**
2. Several **"shipped / compliant" claims do not hold at the code level** ‚Äî SIN encryption, idempotency, consent immutability, and per-product scoring config are enforced on paper or in app-code only, not where they were claimed.
3. The **scorecard scoring model is not in the codebase at all yet** ‚Äî the committed engine uses the raw bureau score + threshold bands. The "clamping" concern lives in the scorecard *design document*, not this code.

**Test suite:** `198 passed, 1 skipped, 239 errored` ‚Äî every error is a no-local-Postgres fixture failure; **zero real test failures**. DB-backed behavior could not be exercised locally.

Severity legend: **CRITICAL** = launch-blocking / active exposure ¬∑ **HIGH** = must-fix before real lending ¬∑ **MED** = correctness/robustness ¬∑ **LOW** = hygiene.

## 1. The scorecard / clamping question

- The committed engine (`app/services/flow_engine.py`) takes the **raw bureau score verbatim** (`effective_score = soft.score` / `hard.score`, lines 320/341) and bands it against thresholds. **There is no base-score + bins model and no 350‚Äì850 clamp anywhere** (grep for `350`/`850`/`clamp` in the scoring path returns nothing). The only `min()/max()` calls clamp an unrelated legacy risk-deduction float.
- `config/decision_rules/` and `config/credit_products/` (described in the spec) **do not exist**; there is no `app/services/decision_rules.py`. Product config lives as a JSONB `verification_matrix` column validated by a JSON Schema.
- **The real config problem is the inverse of the clamping fear:** the engine reads a per-product `manual_review_band`, but the JSON Schema sets `additionalProperties: false` and omits it (`app/schemas/credit_product_verification_matrix.json:65`), so any product that sets it is **rejected at save** and every product silently falls back to a **hardcoded `{600, 679}` band** (`flow_engine.py:48,131`). `bureau.min_score` is defined in config but **never read** (`flow_engine.py:217`). `affordability.max_dti` is in the schema but **no DTI logic exists**.

**Takeaway:** the scorecard model has not landed in code. When it does, the wiring path (`verification_matrix` schema ‚áÑ engine reads) must be fixed so thresholds are genuinely config-driven.

## 5. What is genuinely solid (verified)

- Platform-flow **state machine** ‚Äî transitions only via the engine/orchestrator (Hard Rule #5 holds for `platform_credit_applications`).
- Applicant API is **IDOR-safe** ‚Äî object-scope checks on every protected endpoint (`require_app_scope`).
- **Magic-link + patient JWT** fundamentally sound (SHA-256 token storage, single-use via append-only event, `alg` pinned, expiry enforced).
- **Webhook HMAC** verified first, `hmac.compare_digest` everywhere, no bypass ‚Äî Didit (X-Signature-V2 + timestamp window), Flinks (authenticity-key), Twilio (`RequestValidator`), Resend/Svix all correct.
- **No PII or SIN in logs** ‚Äî explicit redaction (`_redact_pii`); SQL params not logged.
- **No hardcoded secrets**; prod startup validator is real and loud (fails when `USE_REAL_*` flags are on without creds).
- **Notifications genuinely real** (Resend + Twilio) and correctly flag-gated; two-step write rolls back cleanly (no orphan events); suppression gate correct.
- **Migration chain is clean & linear** (the spec's "duplicate migration" was a doc artifact); `platform_events` WORM trigger is solid.
- **PostHog bridge is PII-clean** (hashed IDs, scalar allowlist; no `rich_payload`/`before`/`after`).
- Real **Didit + Flinks** adapters (initiate + webhook result) are genuinely implemented and a true config-only flag swap; a real **Equifax + TransUnion** HTTP client also exists (was reachable only via the now-removed `/credit` endpoint ‚Äî needs re-wiring into the flow as a config swap).

## 6. Remediation already applied ‚Äî P8.1 (this branch)

**`feature/p8.1-security-hardening`** ‚Äî un-mounted the entire unauthenticated legacy V1 lending surface (C-1, C-2), mirroring the P7.1 / P7.4c "un-mount first, delete in a follow-up commit" pattern:

- `app/api/v1/api.py` ‚Äî removed `loan, credit, underwriting, funding, vendor, analytics` router includes + imports. Retained live V2 surface: `auth, credit_products, patients, health`.
- `app/api/v1/endpoints/__init__.py` ‚Äî stopped eagerly importing the un-mounted modules (the P7.1 shared-import trap) ‚Äî reduced to `auth, health`.

**Verification:** app imports cleanly; all six surfaces now 404; 26 live V2 routes intact; `438 tests collected` with **no import/collection errors**; `198 passed` (parity with baseline). No test exercised the removed routes.

**Follow-up (commit B, not yet done):** delete the endpoint files, their schemas (`schemas/{loan,funding,underwriting,vendor,credit}`), dead services (`risk_engine.py`, `credit_bureau.py` ‚Äî but re-home the real Equifax/TransUnion client first), and legacy models (`models/{loan,credit,kyc}`); update `app/models/__init__.py`. None are referenced outside the removed surface + their own tests.

## 7. Compliance gaps for a regulated lending launch

- **Adverse-action (ECOA/FCRA) notice: absent.** Declined applicants receive no notice. Was "out of scope" for the MVP sprint; required before real lending. (`flow_orchestrator._decide` emits `decision_made` but triggers no notice.)
- **SIN encryption-at-rest: unimplemented** (C-3).
- **Quebec block is inert in the live path** ‚Äî `province=None` hardcoded in the orchestrator, so the QC gate never fires.
